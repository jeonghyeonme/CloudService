# 🚀 스마트 스터디 메신저

AWS 서버리스 아키텍처를 기반으로 하는 실시간 웹 메신저 프로젝트입니다. React와 Node.js/Python을 사용하여 스터디 그룹을 위한 효율적인 커뮤니케이션 및 자료 공유 기능을 제공하는 것을 목표로 합니다.

## 📁 디렉토리 구조 (Frontend)

프론트엔드(`frontend/`) 디렉토리의 구조와 각 폴더의 역할은 다음과 같습니다.

```
frontend/
└── src/
    ├── assets/       # 이미지, 폰트 등 정적 에셋
    ├── components/   # 재사용 가능한 공통 UI 컴포넌트 (e.g., Button, Modal)
    │   └── layout/   # 페이지 레이아웃 관련 컴포넌트 (e.g., Header, Sidebar)
    ├── features/     # 기능별 비즈니스 로직 및 컴포넌트 (e.g., auth, chat)
    ├── hooks/        # 여러 기능에서 공통으로 사용하는 커스텀 훅
    ├── lib/          # API 클라이언트, 유틸리티 함수 등
    ├── pages/        # 최상위 페이지 단위 컴포넌트
    ├── providers/    # 전역 상태 관리(Context) 프로바이더
    └── styles/       # 전역 스타일 및 테마
```

-   **`src/assets`**: 로고 이미지, 아이콘, 폰트 등 코드에서 직접 참조하는 정적 파일을 저장합니다.
-   **`src/components`**: 특정 기능에 종속되지 않는, 프로젝트 전반에서 재사용되는 UI 컴포넌트(예: `Button`, `Modal`, `Input`)와 레이아웃 컴포넌트(`Header`, `Sidebar`)를 관리합니다.
-   **`src/features`**: 인증, 채팅, 파일 공유 등 각 기능(도메인)에 특화된 컴포넌트, 비즈니스 로직(Hooks), 상태 관리를 모아둡니다. 가장 핵심적인 비즈니스 로직이 이곳에 위치합니다.
-   **`src/hooks`**: 여러 기능에서 공통으로 사용될 수 있는 커스텀 훅(예: `useWindowSize`, `useDebounce`)을 관리합니다.
-   **`src/lib`**: 외부 API 연동 클라이언트, 날짜 포맷팅과 같은 순수 유틸리티 함수, 상수 값 등을 저장합니다.
-   **`src/pages`**: 사용자가 브라우저에서 특정 경로로 접근했을 때 보게 되는 페이지 단위의 컴포넌트입니다. 여러 `features`와 `components`를 조합하여 하나의 완전한 페이지를 구성합니다.
-   **`src/providers`**: React Context API 등을 사용하여 인증 정보, 테마 등 애플리케이션의 전역 상태를 관리하고 하위 컴포넌트에 제공하는 역할을 합니다.
-   **`src/styles`**: `index.css`와 같은 전역 CSS 파일, 스타일 변수, 테마 설정 등을 관리합니다.

# 📌 프로젝트 그라운드 룰 (Ground Rules)

성공적인 프로젝트 수행과 원활한 협업을 위해 우리 팀이 반드시 지켜야 할 최소한의 규칙입니다. 
코드를 작성하거나 기획을 변경할 때 항상 이 문서를 참고해 주세요.

## 🛠️ 1. 기술 스택 (Tech Stack)

우리 프로젝트는 **AWS 서버리스(Serverless) 아키텍처**를 기반으로 작동합니다. 무거운 서버 환경 대신 클라우드 네이티브 기술을 적극 활용합니다.

- **프론트엔드 (Frontend):** `React` 또는 `Vue 3` (Amazon S3 정적 웹 호스팅 + CloudFront 배포)
- **백엔드 (Backend):** `Node.js` 또는 `Python` (AWS Lambda + API Gateway)
- **데이터베이스 (DB):** `Amazon DynamoDB` (NoSQL)
- **스타일링 (UI):** `Tailwind CSS` + 디자인 컴포넌트 라이브러리

## 🌳 2. Git 브랜치 전략 (Git Flow 기반)

안정적인 CI/CD 환경 구축을 위해 브랜치는 크게 3가지 구조(main, dev, feature)로 나누어 관리합니다.

## 📍 브랜치 역할

- 👑 **`main` (최종 배포용):** 실제 서비스 환경에 배포할 에러 없는 완성된 코드만 관리합니다. (직접 Push 절대 금지 ❌)
- 🛠️ **`dev` (개발 및 통합 테스트용):** 각자 개발한 기능들을 병합하고 통합 테스트를 진행합니다.
- 🌿 **`feature/기능명` (개인 작업용):** 개별 기능 개발을 위해 `dev` 브랜치에서 분기하여 작업을 수행합니다. (예: `feature/login`, `feature/chat-ui`)

## 🔄 작업 및 병합(Merge) 프로세스

1. **작업 시작:** 새로운 기능을 개발할 때는 항상 최신 상태의 `dev` 브랜치에서 `feature/기능명` 브랜치를 생성합니다.
2. **기능 완료 (`feature` ➡️ `dev`):** 작업이 끝나면 GitHub에서 `dev` 브랜치로 **Pull Request (PR)**를 생성합니다.
3. **코드 리뷰 (필수):** 팀원의 PR을 확인하고, 최소 1명 이상의 승인(Approve)을 받아야 `dev`에 Merge 할 수 있습니다.
4. **최종 배포 (`dev` ➡️ `main`):** `dev` 브랜치에서 통합 테스트를 거쳐 버그가 없음을 확인하면, `main`으로 통째로 Merge 하여 실제 AWS 운영 환경에 배포합니다.

## 📝 3. 커밋 메시지 컨벤션 (Commit Convention)

버그 추적과 코드 리뷰를 쉽게 하기 위해 커밋 메시지는 **반드시 정해진 말머리(Prefix)**를 사용합니다.

- `Feat:` 새로운 기능 추가 (예: `Feat: 소셜 로그인 UI 구현`)
- `Fix:` 버그 수정 (예: `Fix: 100명 초과 시 입장되는 오류 수정`)
- `Design:` CSS, UI 레이아웃 등 디자인 변경
- `Refactor:` 기능 변화 없이 코드 구조나 성능을 개선 (리팩토링)
- `Docs:` README, 기능 명세서 등 문서 수정
- `Chore:` 패키지/라이브러리 설치, 환경 설정 파일 수정

## ☁️ 4. 클라우드 및 보안 공통 규칙

AWS 인프라를 다루는 만큼 보안과 자원 관리에 각별히 신경 써야 합니다.

- **🚫 API 키 및 비밀번호 하드코딩 절대 금지:**
    - AWS Access Key, DB 접속 정보 등은 **절대 코드에 직접 적지 않습니다.**
    - 반드시 `.env` 파일에 환경 변수로 분리하고, `.gitignore`에 `.env`를 등록하여 GitHub에 올라가지 않도록 합니다. (요금 폭탄 주의!)
- **🏷️ AWS 리소스 네이밍 규칙 통일:**
    - AWS 콘솔에서 리소스를 쉽게 식별할 수 있도록 이름을 통일합니다.
    - 규칙: `프로젝트명-환경-서비스-용도` (예: `studyhub-dev-s3-frontend`)
- **📢 변경 사항 즉각 공유:**
    - 데이터베이스 테이블(DynamoDB) 변수명을 바꾸거나, API 요청/응답 형태를 변경한 경우 작업방(카톡 등)에 **즉시 공유**하여 다른 파트(프론트/백)의 코드가 터지지 않도록 합니다.
